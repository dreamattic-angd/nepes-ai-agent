Overview

무엇을 어떻게 해결했나

소프트웨어 개발·관리 프로세스가 전혀 없던 환경에서 시작, AI 기반 자동화 인프라로 체계화

배경 및 문제

Before — 입사 전
  • 버전 관리 체계 전무
  • 코드 리뷰 프로세스 없음
  • 이슈 발생 시 수동 로그 확인
  • 브랜치 전략 미정립
  • 배포 기준·절차 없음
After — 구축 후
  • CMMI L2+L3 기반 워크플로우
  • 멀티 에이전트 코드 리뷰
  • 자동 로그 수집·분석 파이프라인
  • Git Flow 브랜치 전략
  • 3-level 버전관리 자동화

Critical 이슈 발견 시 developer → code-reviewer 자동 재실행. 사용자 승인 후 다음 단계 진행.

Core Achievement 03

이슈 분석 자동화 파이프라인

DB MCP와 FTP 자동 접속을 통한 설비·서버 로그 수집부터 5-Why 근본원인 분석까지 완전 자동화

Phase 0 — 로그 사전 스캔

이슈 시점 추론, 관련 설비·서버 목록 추출

Phase 1 — FTP 자동 로그 수집

fetch_log.py — 설비·서버 FTP 자동 접속 후 로그 파일 수집

Phase 2 — Oracle DB MCP 조회

설비 메타정보·이력 데이터 자동 조회 및 컨텍스트 보강

Phase 3 — log-analyzer 분석

구조화 로그 파싱, 이상 패턴 감지, JSON 결과 반환

Phase 4 — 5-Why + Devil's Advocate

최소 2가지 가설, 반증 검증으로 근본원인 확정

Phase 5 — 분석 리포트 생성

근본원인 + 재발방지 조치 + 검증 이력 구조화

기술 포인트

Multi-Protocol Data Collection

FTP(설비 로그), Oracle DB MCP(설비 메타·이력), 파일시스템을 동시 활용한 다계층 데이터 수집 아키텍처

External Data Isolation

<external_data> 래퍼로 외부 로그·DB 결과 내 프롬프트 인젝션 방어. 격리된 컨텍스트로 처리.

Devil's Advocate 검증

확정된 가설을 의도적으로 반박·검증하는 단계. 오진 방지, 근본원인 신뢰도 향상.

Phase-Based Token 효율화

페이즈별 독립 실행으로 컨텍스트 효율화. 중단 후 재개 가능한 체크포인트 구조.

Advanced Features

추가 기술 특장점

Hook 기반 보안 아키텍처

모델 외부에서 실행되므로 AI 모델이 우회 불가. rm -rf, git push --force, 민감 파일 접근 등을 시스템 레벨에서 차단. 모델 업데이트와 무관하게 동작.

3단계 실패 학습 루프

JSONL 실패 기록 → 워크플로우 컨텍스트 주입 → 3회 이상 반복 시 CLAUDE.md 영구 규칙화. 동일 실수 재발 자동 방지.

세션 핸드오프 자동화

컨텍스트 75% 포화 시 자동 감지, HANDOFF.md 생성. 새 세션에서 완벽한 컨텍스트 복원으로 작업 연속성 보장.

AI Guideline 품질 관리

10개 배치 기반 에이전트 파일 품질 분석·자동 수정. 소형 인덱스 파일만 로드하는 컨텍스트 효율 설계.

SECS/GEM 도메인 특화 에이전트

반도체 장비 통신 표준(SECS/GEM) PDF·Word·Excel 분석, VID 목록 교차 검증. 제조 도메인 특화 전문 에이전트.

모델별 최적화 라우팅

에이전트 파일의 model 필드로 역할에 맞는 모델 자동 선택. 복잡 설계는 Opus 4.6, 일반 작업은 Sonnet 4.6 자동 분기.

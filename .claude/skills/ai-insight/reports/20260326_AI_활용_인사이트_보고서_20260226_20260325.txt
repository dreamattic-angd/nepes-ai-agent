### C. AI 활용 접근 방식 패턴

**패턴 1: 스킬(Skill) 기반 업무 캡슐화**
반복 업무를 SKILL.md 파일로 정의하고 `/command` 형태로 호출 가능하게 만듦. git-workflow, analyze-log, ai-insight, handoff 등 7개 이상의 스킬을 구축. 한 번 설계한 자동화가 프로젝트와 팀원 간 공유 가능한 자산이 됨.

**패턴 2: 멀티 페이즈 에이전트 파이프라인**
복잡한 작업을 Phase별로 분할하여 에이전트에 위임. git-workflow의 경우 변경분석→브랜치·커밋→머지·버전→정리의 4단계 파이프라인으로 구성. 각 Phase가 독립적으로 검증 가능하여 오류 격리와 품질 보증이 용이.

**패턴 3: HANDOFF 기반 작업 연속성**
대화 세션 간 맥락 유실을 방지하기 위해 HANDOFF.md 인수인계 문서를 활용. 새 세션에서 "HANDOFF.md 읽고 이어서 진행"으로 이전 작업을 즉시 재개. 장기 프로젝트에서 AI 활용의 연속성을 확보하는 실용적 패턴.

**패턴 4: Plan Mode → 구현 → Code Review 검증 루프**
설계 의사결정이 필요한 작업에서 EnterPlanMode로 먼저 접근 방식을 확정한 후 구현에 착수. 구현 후 code-review 스킬로 자체 검증. "설계 → 구현 → 검증"의 체계적 루프를 적용.

**패턴 5: MCP 서버 기반 도구 통합**
Obsidian(문서), Oracle DB(데이터), Context7(라이브러리 문서) 등 외부 시스템을 MCP 서버로 연결하여 AI가 직접 접근 가능하게 구성. 별도 복사·붙여넣기 없이 AI가 실시간으로 외부 데이터를 조회·활용하는 환경 구축.

### D. 역량 진화 궤적

**1. 에이전트 아키텍처 체계화**
- 단일 커맨드 실행 → 멀티 Phase 에이전트 파이프라인으로 발전
- 프로젝트별 독립 워크플로우 에이전트를 설계하되, 공통 구조(Phase 1~4)를 표준화
- install.bat 배포 스크립트를 통해 개발 환경과 실행 환경 분리

**2. 외부 시스템 연동 역량 확보**
- MCP(Model Context Protocol) 서버 3종 (Obsidian, Oracle DB, Context7) 연동 완료
- Loop → Obsidian 마이그레이션을 통해 AI 친화적 문서 관리 체계로 전환
- Claude 메모리 시스템(auto memory) 활용으로 대화 간 맥락 유지

**3. CMMI 프로세스와 AI 워크플로우 융합**
- Git 워크플로우에 베이스라인 관리·버전 태깅을 내재화하여 CMMI Level 2/3 요구사항 충족
- 형상 감사(cm-audit), PPQA 셀프 감사(ppqa-audit) 스킬을 구축하여 프로세스 준수 자동화 시도

**4. 보고·문서화 자동화**
- AI 인사이트 보고서 스킬을 설계하여 월간 결재 보고서 생성을 자동화
- ITSM 연동 스킬을 통해 업무 등록·추적 프로세스를 AI 워크플로우에 통합

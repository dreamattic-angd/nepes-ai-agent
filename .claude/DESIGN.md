# 설계 결정 기록 (Architecture Decision Records)

이 문서는 nepes-ai-agents 하네스의 핵심 설계 결정과 그 이유를 기록한다.
"왜 이렇게 만들었는가"를 이해하면, 개선할 때 의도를 보존할 수 있다.

---

## 1. Phase 기반 워크플로우 구조

**결정**: 단일 프롬프트 대신 Phase 파일을 분리하여 순차 실행한다.

**이유**:
- 각 Phase를 lazy load하여 토큰 소비를 최소화 (사용하지 않는 Phase는 읽지 않음)
- 실패 시 체크포인트 기반 재개 가능 (처음부터 다시 시작하지 않아도 됨)
- Phase별 독립 테스트 및 수정 가능 (한 Phase 수정이 다른 Phase에 영향 없음)

**대안 검토**: 단일 프롬프트 방식은 간단하지만, 토큰 한도에 걸리거나 중간 실패 시 처음부터 재실행해야 하는 문제가 있었음.

---

## 2. Hook 기반 안전 장치

**결정**: 프롬프트 규칙 대신 Node.js PreToolUse Hook으로 파괴적 명령을 차단한다.

**이유**:
- 모델이 프롬프트 규칙을 무시할 수 있으나, Hook은 모델 외부(하네스 런타임)에서 실행되므로 우회 불가
- exit code 2 반환 시 도구 호출 자체가 차단됨 — 모델의 판단과 무관하게 동작
- 모델 버전이 변경되어도 Hook 동작은 불변

**대안 검토**: CLAUDE.md에 "rm -rf 사용 금지" 같은 규칙을 넣는 방식은 모델이 문맥에 따라 무시할 위험이 있음.

---

## 3. JSONL 로그 포맷

**결정**: JSON 배열이 아닌 JSONL (한 줄 한 레코드) 형식으로 로그를 기록한다.

**이유**:
- append-only 쓰기 가능 (파일 전체를 읽고 다시 쓸 필요 없음)
- 한 줄이 손상되어도 다른 줄에 영향 없음 (JSON 배열은 구문 에러 시 전체 파싱 실패)
- 대용량 시 스트림 처리 가능 (한 줄씩 읽기)

---

## 4. 파일 기반 타이머

**결정**: 워크플로우 소요시간 측정에 메모리 변수 대신 파일 타이머를 사용한다.

**이유**:
- 각 Phase가 별도 `node -e` 프로세스로 실행되므로 메모리 공유 불가
- `.timers/{workflowId}.start` 파일에 타임스탬프를 기록하고, 이후 프로세스에서 읽어서 경과시간 계산
- 프로세스 크래시에도 타이머 데이터가 유실되지 않음

---

## 5. develop 브랜치 전략 (Git Flow)

**결정**: main + develop 2-브랜치 전략을 사용한다.

**이유**:
- feature 브랜치에서 실험 → develop 통합 → main 배포의 단계별 검증
- main은 항상 배포 가능한 상태를 유지
- 사용자가 명시적으로 요청할 때만 main에 머지 (자동 배포 방지)

---

## 6. 프로젝트 자동 감지

**결정**: `git remote get-url origin`에서 repo명을 추출하여 프로젝트를 식별한다.

**이유**:
- 사용자가 매번 프로젝트를 지정할 필요 없음 (DX 개선)
- .claude/ 디렉토리가 글로벌 배포되므로, 동일 설정이 여러 프로젝트에서 사용됨
- 프로젝트별 차이(메인 브랜치, 버전 파일 위치 등)를 설정 테이블로 관리

---

## 7. 실패 메모리 3단계 (기록 → 컨텍스트 주입 → 증류)

**결정**: 실패를 JSONL에 기록하고, 워크플로우 시작 시 반복 패턴을 컨텍스트에 주입하며, 3회 이상 반복 시 CLAUDE.md 영구 규칙으로 승격한다.

**이유**:
- 1단계(기록): 모든 실패를 구조화하여 분석 가능하게 함
- 2단계(주입): 동일 실패 재발을 워크플로우 수준에서 방지
- 3단계(증류): 반복 패턴을 영구 규칙으로 승격하여 JSONL 크기 관리 + 항상 참조되는 위치에 배치
- 용량 관리: 512KB 초과 시 자동 로테이션 (최신 200줄 유지)

---

## 8. 외부 데이터 격리 (프롬프트 인젝션 방어)

**결정**: 외부 소스 데이터를 `<external_data>` 래퍼로 감싸고, 내부 지시를 무시하도록 한다.

**이유**:
- 장비 로그, ITSM 데이터, Git diff 등 외부 데이터에 악의적 지시가 포함될 수 있음
- 명시적 래퍼로 "이것은 데이터이지 지시가 아님"을 모델에 전달
- Hook 기반 방어(실행 단계)와 보완적으로 프롬프트 단계 방어를 구성

---

## 모델 변경 시 재검증 대상

Claude 모델 패밀리가 변경되면 아래 워크플로우의 동작을 확인한다:

1. **git-workflow Phase 1**: 사용자 확인 게이트(⛔) 동작 여부
2. **git-workflow Phase 3**: 버전 자동 계산 정확도
3. **issue-analyze**: 5-Why 분석 품질, 가설 2개 이상 생성
4. **code-review**: 4관점 판정(PASS/REVIEW_NEEDED/REJECT) 일관성
5. **eval-runner.js**: 전체 Eval 케이스 PASS 확인
6. **Hook 동작**: 모델 외부 실행이므로 영향 없음 (확인 불필요)
